饿汉式

即在加载这个类的时候就创建出这个单例，并不是在使用这个类的时候创建。

需要提到的是，饿汉式全都是在加载过程的初始化阶段进行初始化的，在初始化阶段，调用的是类构造器 < clinit >()，JVM会保证一个类的类构造器在多线程环境中被正确的加锁、同步，如果多个线程同时去初始化一个类，那么只会有一个线程执行()方法，其他的线程都要阻塞。注意到，如果一个线程的（）方法中有很耗时的操作，就可能造成多个线程阻塞甚至死锁，而这种问题常常比较隐蔽，所以当我们在静态代码块中写一些语句的时候，包括第三种方法，尽量的避免做耗时的操作。

静态代码块

要知道，加载类的时候，有一个步骤为初始化，在初始化阶段会收集类中的所有类变量的赋值动作和静态代码块中的语句。并执行，所以我们也可以在静态代码块中初始化单例类，也是属于饿汉式的。这种情况适合一些需要复杂的初始化的场景。

懒汉式

即我们需要用到这个对象的时候，才去创建这个对象。也就是说，对象的创建不是在类加载的时候去创建了。

线程不安全

最基本的一中，就是调用方法获取单例对象，调用这个方法的时候我们再去创建这个单例对象。

以下的几种都是线程安全的

静态内部类

解释以下为什么要用一个静态内部类包装一下，在加载Singleton类的时候， 其静态内部类并不会被加载，当外部有调用getInstance()方法的时候，才加载SingletonHolder类，这个时候的过程就跟饿汉式的第一种一样了，是调用（）方法去创建 Singleton 对象的，虚拟机保证了其线程安全性。

加锁

这种加锁太粗暴，就是当每一次获取单例的时候都要走加锁的步骤，就算单例已经创建出来了。当单例创建出来之后，就只用返回单例对象就行了，这种只有读的时候，线程是安全的，不用加锁，所以引出下面的双重校验锁来实现单例模式

双重校验锁

同时还要加上volatile修饰符，我们知道volatile的作用有两个：1. 保持可见性。2. 防止指令重排，在这里就是为了防止指令重排。

INSTANCE = new Singleton() 这条语句在虚拟机中包括三步：

  1. 为要创建的对象分配内存空间
  2. 初始化对象
  3. 将 Singleton 对象的地址赋值给 INSTANCE 变量

正常的步骤是1，2，3。不会出现什么问题，但是指令重排可能会出现1，3，2的步骤。当执行完 3 时，INSTANCE变量已经指向一个地址了，不为null，这个时候新线程来了，判断INSTANCE不为空，直接就返回了未完全初始化的内存的地址，可能会报异常，所以要防止这个指令重排。
